# `/build` コマンド追加 - resonite-headless-container のイメージビルドをKick

## 概要

Discord から GitHub Actions の `workflow_dispatch` をトリガーし、`pluser/resonite-headless-container` のDockerイメージビルドを開始する `/build` スラッシュコマンドを追加する。

## 設計判断

| 項目 | 決定 |
|------|------|
| コマンド構造 | 新しいトップレベル `/build` コマンド |
| 対象リポジトリ | `pluser/resonite-headless-container`（設定で変更可） |
| ビルド対象ref | オプション指定可（デフォルト: `main`） |
| HTTP クライアント | Node 22 ネイティブの `fetch`（依存追加なし） |
| 認証 | GitHub Personal Access Token (PAT) を config で管理 |

## 変更ファイル一覧

### 1. `src/config.ts` - 設定スキーマ拡張

- `githubToken: string` を追加（必須）
- `buildRepo: string` を追加（オプショナル、デフォルト: `"pluser/resonite-headless-container"`）

### 2. `src/github.ts` - 新規作成

GitHub API を呼び出すモジュール:

```typescript
export interface TriggerBuildResult {
  success: boolean;
  message: string;
  runUrl?: string;
}

export async function triggerWorkflowDispatch(
  owner: string,
  repo: string,
  workflowId: string,
  ref: string,
  token: string,
  inputs?: Record<string, string>,
): Promise<TriggerBuildResult>
```

- `POST /repos/{owner}/{repo}/actions/workflows/{workflow_id}/dispatches` を呼び出す
- workflow_id は最初の實装では `"docker.yml"` または `"build.yml"` 等の固定値ではなく、設定オプションとする（`buildWorkflow` config field）
- エラーハンドリング（404, 422, 認証エラー等）

### 3. `src/commands.ts` - コマンド定義とハンドラ追加

**コマンド定義:**

```
/build [ref:<branch_or_tag>]
```

- `ref` オプション（オプショナル、デフォルト: `"main"`）
- 既存の `isAuthorized()` で認可チェック（サービス管理と同じ権限モデル）

**ハンドラ `handleBuildCommand`:**

1. 認証チェック
2. `deferReply()` で応答を遅延
3. config から `githubToken`, `buildRepo` を取得
4. `triggerWorkflowDispatch()` を呼び出し
5. 結果を Embed で返却（成功時は run URL 含む）

### 4. `src/index.ts` - 新規コマンドのルーティング

- `buildCommands()` の戻り値に `/build` コマンドを追加（自動反映）
- `interactionCreate` ハンドラに `commandName === "build"` の分岐を追加
- `handleBuildCommand()` をインポートして呼び出し

### 5. `config.example.json` - 設定例の更新

```json
{
  "githubToken": "ghp_YOUR_GITHUB_PAT",
  "buildRepo": "pluser/resonite-headless-container",
  "buildWorkflow": "docker.yml",
  ...existing fields...
}
```

### 6. `config.ts` - スキーマに `buildWorkflow` も追加

- `buildWorkflow: string`（オプショナル、デフォルト: `"docker.yml"`）

### 7. `src/github.test.ts` - 新規作成

- `triggerWorkflowDispatch()` の単体テスト
- fetch をモックして成功/失敗ケースをテスト

### 8. `src/commands.test.ts` - テスト追加

- `/build` コマンド定義のテスト
- `handleBuildCommand()` のテスト（モック使用）

## 実装順序

1. `src/config.ts` のスキーマ拡張
2. `src/github.ts` の新規作成
3. `src/commands.ts` に `/build` コマンド定義とハンドラを追加
4. `src/index.ts` のルーティング更新
5. `config.example.json` の更新
6. テストの追加・実行
7. ビルド確認 (`npm run build`)

## 注意点

- GitHub PAT は `repo` スコープが必要
- `workflow_dispatch` は API 応答が 204 No Content のため、run URL の取得には追加 API コールが必要になる場合がある
- Discord の embed でビルド開始結果をリッチに表示（成功/失敗、トリガーユーザー、ref、リポジトリ名）
